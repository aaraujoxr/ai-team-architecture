Agents as team members, not repo tools.

THE DEFAULT

What repo-scoped agents give you

Open a repo. Start a session. The agent reasons about architecture, writes code, catches bugs. Within that scope, it's genuinely capable.

But it's bounded
One repo. One session. One cognitive mode at a time. It knows the code but not the work — not what phase you're in, not what was tried before, not how you think.
This works. Until —
You need coordination across repos. Persistent knowledge across sessions. Rigor that doesn't degrade as alignment deepens.
THE SHIFT

The difference between a tool and a team member is state.

A repo-scoped agent is stateless relative to the team. It knows the code but not the work. A role-scoped agent carries coordination state, accumulated knowledge, and context about you. Each session builds on the last.

Scope the agent to a role, not a repo. Everything else follows.

WHAT CHANGES

Three structural benefits

01
Knowledge compounds
Cross-cutting patterns, failure modes, integration conventions — accumulated across sessions and repos. The agent gets smarter over time.
02
Context doesn't collapse
Separate functions get separate context boundaries. Deep alignment for design. Fresh eyes for review. No single session holds all modes at once.
03
Coordination scales
A shared filesystem becomes the protocol. Specs, findings, reports — all files on disk. Observable, auditable, persistent. Nothing hidden in expired sessions.
WHY THIS WORKS

The git parallel

You could achieve all three manually. Maintain markdown files. Copy context into each session. Relay status between roles by hand.

People did this with version control too. Git didn't make version control possible — it made it systematic enough that it actually happens.

This architecture does the same for agent coordination. Agents read the ledger because their instructions say to. They write knowledge back because the role definition includes it. The discipline is structural, not aspirational.

ARCHITECTURE

Three principles

⟷
Transversal coordination
Agents share context across repo boundaries through a structured filesystem. Knowledge is stratified by where it's consumed.
Role separation · Filesystem as protocol · Knowledge layering
◉
Statefulness
Coordination requires dynamic state that evolves with the work. The ledger tracks where you are, what's active, what's blocked — not just what's been produced.
Coordination state (the ledger)
⊕
Human coordination
The human routes all information between roles and provides the context that makes every session productive. No agent-to-agent communication.

COORDINATION STATE

The ledger

Artifacts are produced once. The ledger is state — it evolves continuously. The only file that answers: where are we now?

Every role reads it to orient. Multiple projects write to it. The human remains the authority — but isn't the only one keeping it current.

HUMAN CONTEXT

The agent models you, not just the work

The collaboration_context.md encodes how you reason, what formality you expect, when to ask versus proceed, how you handle disagreement. It's the set of conventions a new team member would take weeks to absorb.

WITHOUT IT
Each role infers your preferences independently. Drift across roles. Research gets your style right, but dev sounds like a stranger. Every new session re-learns what the last one figured out.
WITH IT
Every refinement benefits every future session with every role. A new role added six months from now reads the file and immediately knows how to work with you.

Sits in shared/ because it applies to all roles. Role-specific conventions belong in each role's instructions. The routing protocol defines what flows where. The collaboration context defines how to work with the human.

DAY TO DAY

Each session consumes and produces context

ORIENT
Read state
Ledger, collaboration context, knowledge directory. The agent knows what's active, what phase it's in, what's open.
WORK
Execute
Navigate to repos, implement from specs, run tests. Write questions to disk when the spec is ambiguous.
UPDATE
Write state
Update the ledger with phase transitions, branches created, questions surfaced, results produced.
CAPTURE
Persist knowledge
Cross-repo patterns, conventions, failure modes, decisions. The knowledge directory grows into a reference manual.

By session five or six, the agent anticipates integration issues before you raise them.

THE VISION

Four stages of agent state

1
Artifact state
Agents read and write files. Communication through static documents.
FOUNDATION
2
Coordination state
A shared ledger tracks where things are. Agents read it to orient, update it when state changes.
WE ARE HERE
3
Workflow state
Git operations tie to ledger phases. The agent creates branches when phases start, merges when they complete.
NEXT
4
Internalized state
The human discusses intent. The agent handles the coordination mechanics.
ASPIRATION

The ledger is the pivot at every stage — from a file the human maintains, to the pattern from which agents learn the workflow.

OUR SETUP

The quant team at Reya

RESEARCH
Design & spec
Co-develops formal specs — margin models, funding rates, risk parameters. Outputs self-contained deliverables. Runs in Cowork (filesystem access, no terminal needed).
DEV (TRANSVERSAL)
Implementation
Translates specs to code across reya-quant and reya-data. Surfaces questions, accumulates knowledge. Runs in Windsurf via @quant-dev.
AUDIT
Cold review
Reads specs and implementation reports with no prior context. Identifies failure modes, numerical instabilities, parameter sensitivities. Runs in Cowork.

This is one instantiation. You might split by domain instead of function, add a documentation role, or separate research from design. The architecture is the filesystem, the ledger, and the human context — the roles are yours to define.

GETTING STARTED

Setting up your own

01
Define roles
Separate the cognitive modes: design, implementation, review. Map each to a tool with the right access level.
02
Build the filesystem
Create the directory structure: role directories, shared state, routing protocol, collaboration context.
03
Write role instructions
One canonical file per role. What it reads, what it writes, when it updates the ledger, what knowledge to capture.
04
Initialize the ledger
Start with your current workstreams, phases, open questions. Doesn't need to be comprehensive — needs to be accurate.

Different tools for different roles. One shared filesystem. One ledger. One coordination protocol.
