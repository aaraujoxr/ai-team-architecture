Architecture

Agents as team members, not repo tools.
A coordination framework grounded in design, not tooling.
Artur Ferreira de Araujo · 2026
01 / THE DEFAULT
Agents are capable within one scope. But work isn't one scope.
Open a repo. Start a session. The agent reasons about architecture, writes code, catches bugs. Within that scope, it works.
But real work is split — across repos, directories, and tools. Design in one place, implementation in another, review elsewhere. Each agent sees its slice. None sees the work.
The problem isn't capability. It's that context is fragmented across boundaries no single agent can cross.
02 / THE DESIGN CHOICE
Agents should augment the human, not replace them.
Most designs for AI agents try to get them to talk to each other — unsupervised collaboration, agent-to-agent handoffs, autonomous coordination.
We make the opposite choice. The human routes everything. Not because agents can't coordinate — because when they do, alignment drift becomes unobservable. Agents should potentialize the human's capacity, not substitute for their judgment.
Subsidiarity: agents handle execution within their scope. The human retains coordination authority.
03 / ILLUSTRATION
The quant team
One instantiation. Three roles, three cognitive modes, one human coordinator.
RResearch — design & spec
Co-develops formal specs — margin models, funding rates, risk parameters. Self-contained deliverables. Deep alignment with the human's reasoning.
DDev — transversal implementation
Translates specs to code across multiple repositories. Surfaces integration questions, accumulates cross-repo patterns. Different tool, different context.
AAudit — cold review
Reads specs and implementation with no prior context. Identifies failure modes, numerical instabilities. Isolation is the feature.
Each role sees different repos, uses different tools, carries different knowledge. The human is the only one who sees the full picture.
04 / THE PROBLEM
A team that evolves across projects needs to track state.
Projects start, block on each other, resume. Specs get revised. Implementation surfaces questions that loop back to design. Audits uncover issues that reopen closed phases.
Who knows what? What's active, what's blocked? What was tried and abandoned? What did we learn?
The moment you have multiple agents working across time and projects, you need persistent, structured state that none of them currently maintain.
05 / PRINCIPLES
Three design constraints
Derived from the problem, not from the tooling.
01Transversality
Agents share context across repo and tool boundaries. Knowledge is stratified by where it's consumed — role-specific vs. shared — but nothing is locked inside a single tool or session.
02Statefulness
Coordination requires dynamic state that evolves with the work. Not just what's been produced — where things are, what's blocked, what's next. State that persists across sessions and is readable by every role.
03Subsidiarity
The human routes all information between roles. No agent-to-agent communication. Feedback is discussion — implementation after explicit clearance. Agents execute within scope; the human holds coordination authority.
06 / DECOMPOSITION
Three types of knowledge
Different owners, different lifespans, different access patterns.
01Human collaboration context
How you reason, what formality you expect, how feedback works, what domain context every role needs. Owned by the human. Stable across sessions.
02Agent-specific knowledge
Cross-repo patterns, integration conventions, failure modes, design decisions — accumulated by each role through its work. Grows with every session.
03Team state
Where are we now? Which projects are active, blocked, waiting? What phase is each in? The only knowledge that changes within a session, not just between them.
07 / THE GAP
We need what git is to code — for team knowledge.
Git didn't make version control possible — it made it systematic. Commit, branch, merge: a structured interface so developers focus on code, not versioning mechanics.
You could maintain all three types of knowledge manually. Copy context into each session. Relay status between roles by hand. Keep a spreadsheet of what's active. It works the way ad-hoc versioning worked — until it doesn't.
No structured interface for agent coordination exists. That's the gap this architecture addresses.
08 / ARCHITECTURE
Three-component model
The three types of knowledge, plus the orchestration infrastructure that connects them. Abstract design — the next section instantiates it.
09 / THE MEDIUM
Files are the interface
Every handoff is a file on disk. The routing protocol governs who reads what and who writes where. Nothing hidden in expired sessions.
10 / HUMAN CONTEXT
The collaboration context
Not preferences — the operational system that eliminates re-teaching every session. Every refinement compounds across all roles.
01Analytical defaults
Mechanism-first reasoning, dynamic behavioral adaptation, how problems are stated and decomposed.
02Communication patterns
Writing registers, formality level, when to write versus wait, how documents differ from research chronology.
03Feedback protocol
Feedback is discussion, not instruction. Implementation happens after explicit clearance, not on first reaction.
04Domain context
Sufficient for any role to orient without preamble. A new role added months later reads this file and immediately knows how to work with you.
11 / TEAM STATE
The ledger
A live artifact mapping workstreams and team evolution. Not a static document — it evolves continuously as work progresses.
All roles read it to orient. The human maintains it — agents propose updates. The ledger's schema matters: precise semantics, not just suggested fields.
Project A Spec delivered — phase 1 ready for implementation
Project B Parameter sweep complete — results in shared/reports/
Project A Branch created — feature/project-a-phase1
Project C Migration staged — blocked on Project A
Project A Question surfaced → routed to design
Project B Phase 2 started — review routed
Project A Phase 1 complete — PR merged, unblocks C
Project C Unblocked — phase 1 started …
12 / TOOLING
Tools serve the design, not the other way around
01Thin pointers
Each tool gets minimal config pointing to the filesystem. The canonical role definition is ROLE.md on disk — tool config is just a reference to it.
02Different tools, different modes
IDE-native tools for implementation. Filesystem-native tools for design and review. The architecture doesn't prescribe the tooling — it prescribes the interface.
03MCP as the bridge
Model Context Protocol extends tool capabilities — filesystem access, browser integration, external APIs. Capabilities vary by tool; the filesystem is the constant.
→Knowledge convergence
All role knowledge is filesystem-native: inspectable, versionable, tool-agnostic. One source of truth, loaded from one place, by any tool.
13 / THE FULL PICTURE
The quant team setup
All the pieces in one place. Three roles, three tools, one filesystem, one human routing everything through a shared ledger.
RResearch → Cowork
Filesystem access without terminal. Specs, mechanism design, parameter analysis. Reads collaboration context + own knowledge. Writes specs to shared/reports/.
DDev → Windsurf via skill
IDE-native. A Windsurf skill points to ROLE.md on the shared filesystem — full role context from a single reference. Reads specs, writes implementation reports. Cross-repo.
AAudit → Cowork (isolated)
Cold review. No prior context, no accumulated alignment. Reads specs and implementation reports. Writes audit findings. Isolation is structurally enforced.
→The routing protocol
Human routes specs from Research → Dev. Routes implementation reports → Audit. Routes audit findings back. The ledger tracks it all. Every handoff is a file.
14 / DAY TO DAY
Each session consumes and produces context
Orient
Read state
Ledger, collaboration context, knowledge directory. The agent knows what's active.
Work
Execute
Navigate to repos, implement from specs, run tests. Write questions to disk.
Update
Write state
Ledger updates: phase transitions, branches, questions surfaced.
Capture
Persist knowledge
Cross-repo patterns, conventions, failure modes, decisions.
By session five or six, the agent anticipates integration issues before you raise them.

16 / ACTIVE DEVELOPMENT
This is work in progress.
The architecture, the tooling, and the coordination patterns described here are under active development. The framework is being refined through real use — every session surfaces new learnings and edge cases.
