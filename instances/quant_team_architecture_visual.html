Routing Topology
Handoff Flows
Research → Dev (spec handoff)
  1. Research writes spec to research/specs/
  2. Artur reviews and decides spec is implementation-ready
  3. Artur opens Dev session, points to spec file path
  4. Dev reads spec as source of truth, implements
Dev → Research (implementation question)
  1. Dev encounters ambiguity in spec during implementation
  2. Dev writes question to dev/questions/ with full context
  3. Artur reads question, decides it's a research question
  4. Artur opens Research session, points to question file
Research → Audit (audit request)
  1. Spec or implementation reaches ship-ready state
  2. Artur decides it's stable enough for critical review
  3. Artur opens Audit session, points to deliverable path(s)
  4. Audit reads deliverable cold — no prior context
Audit → Research (finding response)
  1. Audit writes finding to audit/findings/ with severity
  2. Artur reads audit report, identifies findings needing response
  3. Artur opens Research session, points to specific finding
  4. Research addresses finding within their conversation

Architectural Protection Rules
Immutable Infrastructure

The three-component skeleton is fixed. Agents operate freely within their designated areas but cannot modify the infrastructure that defines how the team works.

No agent may modify ledger.md — agents nudge Artur when it's stale
No agent may modify collaboration_context.md
No agent may modify routing_protocol.md
No agent may write outside its designated write scope
No agent may read or explore directories outside its access scope
Why these rules exist

The threat model is drift and confusion, not adversarial attack. Agents aren't trying to break things — they lose track of boundaries over long sessions, especially when context is large. The rules prevent:

  • Alignment drift when roles read each other's reasoning
  • Ledger corruption from well-intentioned but incorrect updates
  • Scope creep when agents "helpfully" fix things outside their domain
How they're enforced today
  • Instruction-level constraints — explicit scope tables and blanket negative constraints in every role's project instructions
  • Routing protocol reference — the access matrix is the canonical source; if a path isn't listed, it's off-limits
  • Git safety net — all changes are version-controlled; unauthorized modifications are reversible
Future: per-project MCP scoping could enforce boundaries at the tool level rather than instruction level.
Cognitive Mode Separation

Role boundaries exist to prevent alignment drift, not security breaches. Each role operates in a distinct cognitive mode:

Research
Deep alignment mode. Co-develops reasoning with Artur. Knows the "why" behind every decision.
Dev
Practical implementation mode. Sees specs cold. Treats them as source of truth without questioning the model.
Audit
Adversarial review mode. Sees deliverables cold. No alignment with the reasoning that produced them.
