AI Team Architecture
Agents as Team Members, Not Repo Tools

A design pattern for role-scoped, stateful AI agents that coordinate through a shared filesystem and evolve from tools into team members.

View Presentation ↗

1. The premise

The default way to use an AI agent in engineering is straightforward: open a repo, start a session, get work done. The agent is scoped to the codebase it can see. Within that scope, it's genuinely capable — it reasons about architecture, writes working code, catches bugs. But it's bounded by what it can see: one repo, one session, one cognitive mode at a time.

This works. Until you need coordination across repos. Until you need persistent knowledge across sessions. Until you need rigor that doesn't degrade as alignment deepens.

The alternative: scope the agent to a role, not a repo.

The difference between a tool and a team member is state. A repo-scoped agent is stateless relative to the team. It knows the code but not the work — not what phase you're in, not what was tried before, not how you think. A role-scoped agent carries state: coordination state about where the project stands, accumulated knowledge about what patterns have emerged, and context about the human it's working with. Each session builds on the last.

This is not a marginal improvement. It changes what the agent can do for you.

The trajectory

The architecture describes a progression in what kind of state agents carry:

1
Artifact state
Agents read and write files. Communication is through static, point-in-time documents — specs, reports, findings.
FOUNDATION
2
Coordination state
A shared ledger tracks where things are: active workstreams, current phases, open questions, blocking dependencies. The system becomes dynamic.
WE ARE HERE
3
Workflow state
Git operations tie to ledger phases. The agent creates branches when phases start, merges when they complete.
NEXT
4
Internalized state
The agent has learned the orchestration patterns deeply enough that the human discusses intent, and the agent surfaces the coordination decisions at the natural points.
ASPIRATION

We operate today between stages one and two. The architecture is designed to support the full progression.

Why this matters

Three structural benefits follow from scoping agents to roles:

Knowledge compounds. A repo-scoped agent starts fresh every session, or at best retains what's in the codebase. A role-scoped agent accumulates cross-cutting knowledge — how data flows between services, what patterns emerged from past implementations, which failure modes were encountered and resolved. This knowledge lives with the agent, not with any single repo. Each session builds on the last.

Context doesn't collapse. When you separate design from implementation from review, each function operates with the context boundary it needs. The design agent builds deep alignment with you over conversation. The implementation agent works from the spec, not the conversation. The review agent sees deliverables cold. No single session tries to hold all three cognitive modes at once.

Coordination scales. A shared filesystem becomes the communication protocol. Specs, questions, findings, reports — they're all files on disk. The human routes between roles by deciding which artifact goes where. The ledger tracks what's active, what's blocked, what's done. This is observable, auditable, and persistent. Nothing is hidden in a conversation that expired.

In theory, you can achieve compounding knowledge, stable context, and scalable coordination manually. Maintain markdown files with accumulated knowledge. Copy relevant context into each session. Write down what phase you're in. Relay status between roles by hand. People do this — it's the engineering equivalent of copying folders and naming them project_v2_final_FINAL.

Git didn't make version control possible. People versioned their code before git existed. What git did is make version control systematic enough that it actually happens — the discipline becomes embedded in the workflow rather than depending on the human remembering to do it.

This architecture does the same for agent coordination. The agents read the ledger because their instructions say to. They write knowledge back because the role definition includes it. They update state because the workflow demands it. The discipline is structural, not aspirational.

2. Principles

Role separation

A single long-running conversation with one agent produces alignment drift — the agent progressively models your preferences and becomes less likely to challenge the work. This is productive for co-development. It is dangerous for review. You want deep alignment when building and fresh eyes when reviewing, and those are mutually exclusive cognitive modes.

The solution: separate the functions that benefit from alignment from the functions that suffer. The minimum viable separation is design, implementation, and cold review. You can extend from there — domain-specific implementation roles, separate design from research, specialized reviewers. The underlying principle is the same: functions with different context requirements get different instances.

One decision deserves emphasis: the review function must have no access to the design reasoning. If a spec requires background reasoning to seem correct, the spec is under-documented. The review function enforces this precisely because it can't see the reasoning chain.

Filesystem as protocol

Agents communicate through files on a shared filesystem, not through conversation history. Files are persistent, observable, and composable. You can audit the full history of an engagement by reading the directories. Nothing is hidden in a session that expired.

The directory structure is the protocol. Each role reads from defined directories and writes to defined directories. A routing document specifies who reads what and who writes where. Each role directory is not just a filing structure — it defines where an agent is spawned, with its own instructions, its own knowledge, and its own output contract:

team-root/ ├── ledger.md # coordination state ├── shared/ │ ├── collaboration_context.md # how to work with the human │ ├── routing_protocol.md # interface contract between roles │ └── reports/ # cross-role deliverables ├── role-a/ # agent spawned here │ ├── CLAUDE.md # role instructions │ ├── knowledge/ # persists across sessions │ └── outputs/ ├── role-b/ # agent spawned here │ ├── CLAUDE.md │ ├── knowledge/ │ └── outputs/ └── role-n/ # agent spawned here └── ...

This is an example, not a prescription. The structure should reflect the team's actual roles and communication patterns. What matters is the principle: each directory is an agent's home, and the filesystem is the communication layer between them.

Knowledge layering

Knowledge lives at the level where it's consumed.

Repo-specific knowledge stays in the repo. Docstrings, READMEs, CI configuration — these belong with the codebase.

Cross-cutting knowledge lives with the agent. When an agent works transversally across repos, it accumulates knowledge that doesn't belong to any single codebase: how data flows between services, what integration patterns emerged, what decisions were made during implementation that aren't in any spec. This knowledge is structurally homeless in a repo-scoped model — it ends up in Slack, in someone's head, or nowhere. In a role-scoped model, it has a natural home: the agent's persistent knowledge directory, loaded at the start of every session regardless of which repo is open.

Team coordination lives in shared state. The routing protocol, the collaboration context, the ledger — read by all roles.

Role instructions are the single source of truth per role. One canonical file defines what the agent does, how it orients, what it writes and where.

This creates a clean separation: what belongs to the code travels with the code; what belongs to the team persists independently.

Coordination state

The ledger is the coordination state for the entire team. It tracks active workstreams, current phases, open questions, blocking dependencies. Every role reads it to orient at the start of a session.

The ledger is different in kind from every other file in the system. Specs, questions, findings, reports are artifacts — produced once, consumed by whoever reads them. The ledger is state. It evolves continuously. It's the only file that answers "where are we right now?"

Project A
Spec delivered — phase 1 ready for implementation
Project B
Parameter sweep complete — results in shared/reports/
Project A
Branch created: feature/project-a-phase1
Project C
Migration staged — blocked on Project A phase 1
Project A
Question surfaced → routed to design
Project B
Phase 2 started — review routed for phase 1
Project A
Phase 1 complete — PR merged, unblocks Project C
Project C
Unblocked — phase 1 started …

This is not just an agent coordination tool. It is how you organize your own timelines and work. The ledger connects your project management to your agent orchestration. Maintaining the ledger is maintaining clarity about the state of the work.

Agents update it when meaningful state changes occur — phase transitions, branches created or merged, PRs opened or closed, questions resolved, implementation results that affect design decisions. The human remains the authority, but isn't the only one keeping it current.

The ledger also connects to the trajectory in §1. At stage one, it's a file the human maintains. At stage two, agents update it. At stage three, it synchronizes with git. At stage four, it becomes the pattern from which agents learn the workflow. Ledger discipline is not overhead. It is the training signal for increasingly autonomous coordination.

Human as router

All routing goes through the human. No agent communicates directly with another.

This is not bureaucracy. It is the mechanism that keeps you central to every decision. When the dev agent surfaces a question about the spec, it writes the question to disk. You read it, decide whether it's a design question or an implementation question, and route it.

Most multi-agent frameworks (CrewAI, AutoGen, LangGraph) focus on autonomous agent-to-agent communication. The failure mode is subtle: context loss and misaligned decisions that compound silently. A human router catches these. The overhead is low — you're already reading the outputs — and the alignment benefit is high. The model is closer to the blackboard architecture from classical AI: agents read from and write to a shared repository, and a human controller decides what gets routed where.

Collaboration context

Agents need to understand the human, not just the work. The collaboration context file encodes operational context: how you reason, what formality you expect, when to ask versus proceed, how you handle disagreements. It's the set of conventions a new team member would take weeks to absorb.

Without it, each role infers your preferences independently — producing drift across roles. A shared collaboration context eliminates this and compounds over time: every refinement benefits every future session with every role. A new role added six months from now reads the file and immediately knows how to work with you.

The collaboration context sits in shared/ because it applies to all roles. Role-specific conventions belong in each role's instructions.

3. Worked example: a quant team at Reya

This section describes how these principles play out in practice with a specific team configuration. The purpose is to make the general architecture concrete — not to prescribe a setup.

The team

Reya's quant team uses three role-scoped agents:

Research — co-develops formal specs (margin models, funding rate mechanisms, risk parameters) with the human. Outputs self-contained specifications that can be implemented without reading the design conversation.

Dev — translates specs into working implementations across multiple repositories. Surfaces practical questions the spec doesn't answer. Maintains knowledge that persists across sessions.

Audit — reads deliverables cold. No access to the design conversation. Identifies failure modes, numerical instabilities, parameter sensitivities. Rates findings by severity. Does not suggest fixes — identifies problems only.

The research agent builds deep alignment with the human on mechanism design — the math, the parametrizations, the tradeoffs. The dev agent works from the spec, not from the design conversation. It encounters practical questions (edge cases, data schema mismatches, performance constraints) and writes them to dev/questions/ for routing. The audit agent sees the spec and the implementation report with no prior context, which is exactly the cognitive state you want for critical review.

The transversal dev agent

This is where "team member, not repo tool" is most visible. The dev agent doesn't live in either repo. It lives in the team's coordination layer and navigates into repos as needed.

The agent is loaded via a Windsurf global skill (@quant-dev) or a Claude Code CLAUDE.md that acts as a thin pointer to the canonical instructions on the shared filesystem. Both entry points converge on the same role definition and knowledge directory. The canonical instructions live in CLAUDE.md on the shared filesystem — not inside the skill file. Changes to the role instructions propagate automatically. Accumulated knowledge in dev/knowledge/ is loaded every session. The agent gets smarter over time without manual context injection. And the entry point can be invoked from any repo: same team member, different codebase.

Tooling choices

Research and audit run in Cowork (Claude Desktop projects). They need filesystem access for reading and writing specs, reports, and findings. They don't need terminal access. Cowork's project system gives them persistent instructions and knowledge files scoped to the role.

Dev runs in Windsurf or Claude Code. It needs terminal access for git, python, psql. It needs to navigate codebases, write implementations, run tests. The global skill system (Windsurf) or CLAUDE.md convention (Claude Code) makes it transversal across repos.

Different tools for different roles, unified by the shared filesystem.

What we learned

The skill has to be invoked with a concrete task. On the first message of a Windsurf session, @quant-dev implement phase 2 of the margin validation works. The skill tag alone sometimes doesn't trigger the file-reading chain reliably.

Knowledge builds slowly but compounds fast. The first few sessions have an empty knowledge directory. By the fifth or sixth session, the agent has accumulated enough cross-repo context that it anticipates integration issues before you raise them.

Cowork project instructions don't auto-propagate. Unlike the dev role (where CLAUDE.md changes propagate through the skill), research and audit instructions are set in the Cowork UI. Changes require manual updates in the project settings.

The ledger catches drift. When the dev agent updates the ledger after completing a phase, the research agent picks it up in its next session and adjusts priorities. Without the ledger, you'd be manually relaying status between roles.

4. Orchestration in practice

The principles describe the architecture. This section describes how the pieces move together in daily use.

The inter-role interface

Roles don't communicate directly. They communicate through a shared filesystem, with the human routing between them. The routing protocol is the contract that defines this interface: what document types flow between which roles, through which directories, and how handoffs are synchronized.

In practice, the flows look like this. Research writes a spec to research/specs/. The human reads it, decides it's ready for implementation, and tells the dev agent where to find it. The dev agent implements from the spec. When it encounters ambiguity, it writes a question to dev/questions/ — not a message to the research agent, but a file on disk that the human can read, contextualize, and route. When implementation is complete, the dev writes a report to shared/reports/. The human routes it to audit. Audit reads the spec and the report cold, writes findings to audit/findings/. The human routes relevant findings back to research or dev.

Every handoff is a file. Every routing decision is the human's. The filesystem makes the communication observable — you can reconstruct the full conversation between roles by reading the directories.

The ledger synchronizes all of this. It's what tells each role, at the start of a session, "here's what's active, here's what's waiting on you, here's what changed since you last looked." Without the ledger, the filesystem is a collection of files. With it, it's a coordination system.

Session workflow

A well-structured session follows a predictable arc:

ORIENT
Read state
Ledger, collaboration context, knowledge directory. The agent knows what's active, what phase it's in, what's open.
WORK
Execute
Navigate to repos, implement from specs, run tests. Write questions to disk when the spec is ambiguous.
UPDATE
Write state
Update the ledger with phase transitions, branches created, questions surfaced, results produced.
CAPTURE
Persist knowledge
Cross-repo patterns, conventions, failure modes, decisions. The knowledge directory grows into a reference manual.

This cycle means each session both consumes and produces context. The agent orients from accumulated knowledge at the start and enriches it at the end.

The knowledge write-back loop

What the agent captures at end of session matters. The rule of thumb:

Goes to the agent's knowledge directory: cross-repo patterns (how data flows from service A to service B), implementation conventions that emerged from spec interpretation, environment quirks, failure modes and resolutions, decisions made during implementation that aren't documented in any spec.

Stays in the repo: code documentation, docstrings, inline comments, CI/CD configuration, repo-specific tooling setup.

Goes to shared/reports/: implementation reports, validation results, review guides — anything that needs to reach other roles.

Use descriptive filenames. Append to existing files when the topic already has one rather than creating duplicates. The knowledge directory should read like a reference manual over time, not a collection of session logs.

Toward implicit orchestration

The session workflow above is largely explicit: the agent reads the ledger because its instructions tell it to, updates state because the role definition says so, captures knowledge because the human asked for it.

The natural endpoint of this architecture is implicit orchestration. If the agent has seen enough sessions, it learns the patterns: "we just finished validation, the next step is to merge the branch and update the ledger." "This implementation surfaced a question about the spec — I should write it to dev/questions/ and flag it." "The knowledge I just accumulated about the data pipeline integration should be written back before this session ends."

This is the progression from stage two to stage four described in §1. The agent internalizes the workflow deeply enough that the human discusses the work, and the agent handles the coordination mechanics.

The difficulties are real. Implicit orchestration is only as good as the state it relies on. If the ledger is stale, the agent orients to a world that no longer exists. If the knowledge directory has contradictory entries, the agent inherits confusion. The architecture demands increasing discipline as it grants increasing autonomy — stage one tolerates a messy filesystem; stage four does not.

This is worth being honest about. The vision is powerful: a human who talks about intent while the agent handles the mechanics of branches, phases, knowledge capture, and routing. But achieving it requires that the foundational layers — the filesystem structure, the ledger, the knowledge directories — are maintained with genuine discipline. The aspiration is real. So is the cost.

5. Setting up your own

Entry points

Two primary entry points can load agents into the role-scoped architecture:

Windsurf global skills — a thin pointer at ~/.windsurf/skills/{skill-name}/SKILL.md that tells the agent where to find its canonical instructions and knowledge on the shared filesystem. Invoked with @skill-name [concrete task]. Ideal for dev agents that need IDE-integrated terminal access.

Claude Code — reads CLAUDE.md natively from the project root or any directory it's pointed at. When CLAUDE.md lives in the role directory on the shared filesystem, Claude Code becomes the entry point for that role. Ideal for terminal-first workflows and agents that benefit from Claude Code's agentic capabilities (subagent spawning, tool use chains).

Both converge on the same canonical role definition. The entry point is a matter of tooling preference — the architecture is the filesystem, the ledger, and the human context underneath.

For roles that don't need terminal access (design, review), Cowork (Claude Desktop projects) provides filesystem access with persistent project-scoped instructions.

MCP configuration

The Model Context Protocol layer gives agents structured access to external tools. For a team architecture, the key MCP servers are:

Filesystem access (@modelcontextprotocol/server-filesystem) — gives the agent read/write access to specified directories. Configure it with paths to both the team coordination directory and any repos the agent needs:

{ "mcpServers": { "filesystem": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/team-root", "/path/to/repo-one", "/path/to/repo-two"] } } }

Terminal access (@wonderwhy-er/desktop-commander) — gives the agent shell access. Essential for dev agents that need git, package managers, test runners, database clients.

The scoping principle: give each role the minimum access it needs. Research and audit typically need filesystem only. Dev needs filesystem plus terminal. Over-permissioning creates risk without benefit.

Step-by-step scaffold

Once you've decided on your role separation:

01
Create the directory structure
Start with the filesystem layout from §2. Adjust the role directories to match your actual functions.
02
Write the routing protocol
A short document that specifies: which role reads from where, which role writes to where, what document types flow between roles.
03
Write the collaboration context
How you reason, communication preferences, what detail you expect, when to ask vs proceed.
04
Write role instructions
One canonical file per role. What it reads, what it writes, when it updates the ledger, what knowledge to capture.
05
Initialize the ledger
Start with your current workstreams, phases, open questions. Doesn't need to be comprehensive — needs to be accurate.
06
Configure entry points
Windsurf skills and/or Claude Code for dev roles. Cowork projects for roles that don't need terminal access. MCP servers for filesystem and terminal.
07
Run a first session
Give each role a concrete task and observe how it orients. The first sessions reveal gaps in instructions and routing.

Diagnostic questions

The right architecture depends on your team's actual constraints. These questions help determine what structure to adopt:

How many repos does your work span? If everything lives in one repo, a transversal agent is unnecessary. If you regularly work across two or more, it becomes essential.

What functions do you need separated? The minimum is design + implementation + review. But you might want domain-specific implementation roles, or separate design from research, or add a documentation role.

Do you need cold review? If you're doing work where rigor matters — financial models, security-critical code, formal specifications — a review function that sees deliverables without design context is valuable. If your work is more exploratory, you might not need it yet.

What tools does each function need? Design and review typically need filesystem access only. Implementation needs terminal access, git, language runtimes. Map the tools to the functions.

Where does your cross-cutting knowledge currently live? If it's in Slack, in someone's head, or nowhere — that's the knowledge a transversal agent will capture. If it's already well-documented in a wiki or shared docs, you might integrate those into the knowledge directory.

How do you currently track project state? The ledger doesn't replace your project management. It complements it by giving agents a structured way to read and update coordination state.
